## Question

独立 Worker 如何每 15 秒读取最小配置、拉取启用币对行情、忽略未收盘 K 线、复用现有公式并发送尚未出现过的两类标记？需要明确最小失败处理、简单状态文件与避免重复发送的规则。

## Resolution

第一版 Worker 是一个单进程、串行的 15 秒循环；它不保存独立队列，也不按 K 线边界另建调度器。这个频率已满足收盘后 10 秒才判定的约束，并将正常首次投递控制在下一轮内（约 25 秒）。

### 每轮扫描

1. 读取并校验 `/data/monitor.json` 的完整快照。文件不存在、`enabled_symbols` 为空时不拉行情，写 `waiting_for_sync` 或 `idle` 状态后等待下一轮。
2. 对每个启用币对，按配置的数据源、周期和最小历史窗口拉取 K 线；适配层统一为按开盘时间升序的规范 Candle，并保留 `close_time_ms`。
3. 用 `now_utc_ms >= close_time_ms + 10_000` 过滤未确认 K 线，再对完整的已收盘历史运行 `jt-regime-v1` 的同一公式和 `buildBottomMarkers()` 语义。不得只计算最新一根，因为标记依赖历史窗口、结构突破和冷却期。
4. 对结果中的 `trial_bottom` 与 `confirmed_bottom` marker，构造 `strategy_version:source:symbol:interval:candle_open_time_ms:signal_type`。只处理**该配置同步时间之后收盘**且不在 `sent_signal_keys` 中的 key；同步后的第一轮因此只建立观察基线，不补发旧图表标记。
5. 依 marker 类型生成当前页面同义的“试探抄底”或“确认抄底”正式消息，调用既有企微发送路径。发送成功后立即将 key 追加到内存集合并原子写回 `/data/monitor.json`；同一根 K 线的两个类型是两个 key，允许各发一次。

### 失败与恢复

- 某币种的行情拉取、计算或企微发送失败，不写 `sent_signal_keys`，本轮继续处理其他币种；失败的信号会在下一次 15 秒循环重试。
- 不设置额外重试队列、指数退避或失败终态；这是最小版本的取舍。企微持续失败会每轮尝试一次，直到成功或配置被关闭。
- Worker 重启后从 `monitor.json` 的 `sent_signal_keys` 恢复去重状态；已成功落键的信号不会重发，未落键的信号会在下一轮重新判断并投递。外部企微成功与文件落键之间崩溃仍可能造成一次重复，沿用 JSON 方案已接受的边界。

### 最小状态文件

每轮开始和结束都原子更新 `/data/worker-status.json`。正常运行时 `state` 为 `running`；无配置为 `waiting_for_sync`，无启用币对为 `idle`；存在本轮局部失败时仍为 `running`，同时记录 `last_error`。每个币种写入最后一个已确认 K 线的开盘时间、最后一次成功发送的 key（如有）和该币种最近错误；下一轮成功后清除相应错误。

```json
{
  "version": 1,
  "updated_at": "2026-07-10T12:00:15Z",
  "state": "running",
  "last_scan_at": "2026-07-10T12:00:15Z",
  "next_scan_at": "2026-07-10T12:00:30Z",
  "last_error": null,
  "symbols": {
    "BTCUSDT": {
      "last_closed_candle_open_time_ms": 1720000000000,
      "last_signal_key": "jt-regime-v1:spot:BTCUSDT:1h:1720000000000:trial_bottom",
      "last_error": null
    }
  }
}
```

### 最小手工验收

1. 以 Docker Compose 启动 Web 与 Worker，并确认 `/data/monitor.json`、`/data/worker-status.json` 都在主机挂载目录出现。
2. 在页面打开当前配置，启用一个币对后点击“一键同步到后台”；状态接口在 15 秒内显示该币种的已确认 K 线，而不是未收盘 K 线。
3. 用固定行情夹具或可控测试数据制造一根在同步后收盘的试探 marker 和一根确认 marker，验证两类各收到一条消息，且 `monitor.json` 增加两个不同 signal key。
4. 保持相同行情再等两个循环，确认不再发送；删除其中一个已发送 key 后重启 Worker，确认该 key 会补发且其他 key 不会重发。
5. 临时令行情或企微请求失败，确认状态文件记录错误、key 未写入；恢复请求后下一轮成功发送并清除错误。
