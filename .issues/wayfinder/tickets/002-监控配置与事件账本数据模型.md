## Question

如何用挂载在 `/data` 的最小 JSON 配置文件保存当前页面的一次性同步结果：启用币对、数据源、周期、公式参数以及已发送信号键？需要规定 Worker 读取、页面写入和容器重启后的简单恢复规则。

## Resolution

范围已收敛为两个挂载 JSON 文件；SQLite、事件账本、历史审计和配置 CRUD API 不进入第一版。

### `/data/monitor.json`

页面通过一次性同步 API 写入。它替代当前 `localStorage` 中与后台运行有关的币对开关、数据源和公式参数；页面布局顺序、自动刷新频率及浏览器的已见标记不写入该文件。

```json
{
  "version": 1,
  "updated_at": "2026-07-10T12:00:00Z",
  "strategy": {
    "key": "jt-regime",
    "version": "jt-regime-v1",
    "source": "spot",
    "interval": "1h",
    "limit": 500,
    "mom_start": 2,
    "mom_end": 52,
    "z_length": 52,
    "ext_thresh": 2.0,
    "smooth_len": 8
  },
  "enabled_symbols": ["BTCUSDT", "ETHUSDT"],
  "sent_signal_keys": [
    "jt-regime-v1:spot:BTCUSDT:1h:1720000000000:trial_bottom"
  ]
}
```

- `enabled_symbols` 是唯一的启停来源：数组内即启用，空数组表示 Worker 空闲。
- 参数范围沿用当前 UI：`limit` 120--1000、`mom_start` 2--50、`mom_end` 3--200、`z_length` 10--200、`ext_thresh` 0.5--5、`smooth_len` 1--80，且 `mom_start < mom_end`；`source` 和 `interval` 限定为当前页面已支持值。
- `sent_signal_keys` 的键格式固定为 `strategy_version:source:symbol:interval:candle_open_time_ms:signal_type`，其中 `signal_type` 为 `trial_bottom` 或 `confirmed_bottom`。成功发送后才追加；保留上限由实现固定为最近 5000 条，超出时删除最旧项。
- 每次同步覆盖 `strategy` 和 `enabled_symbols`，但保留既有 `sent_signal_keys`，从而不因页面保存配置而重复发送已通知的同一信号。策略版本升级时由实现改为新版本字符串，因此自然使用新的去重命名空间。

### `/data/worker-status.json`

仅 Worker 写入，页面只读，用于最小运行状态展示；它不是恢复通知正确性的依据，丢失时可重建。

```json
{
  "version": 1,
  "updated_at": "2026-07-10T12:00:10Z",
  "state": "running",
  "last_scan_at": "2026-07-10T12:00:10Z",
  "next_scan_at": "2026-07-10T13:00:10Z",
  "last_error": null,
  "symbols": {
    "BTCUSDT": {
      "last_closed_candle_open_time_ms": 1720000000000,
      "last_signal_key": "jt-regime-v1:spot:BTCUSDT:1h:1720000000000:trial_bottom",
      "last_error": null
    }
  }
}
```

### 原子读写与重启规则

- Web 与 Worker 对 JSON 都使用“读取完整文件、解析并校验、写入同目录临时文件、`fsync`、原子 `rename`”的替换方式；绝不原地截断写。读到无效 JSON 时保留进程内最后一个有效快照并在 `worker-status.json` 记录错误。
- 同步 API 写 `monitor.json` 时，以文件锁或同进程互斥锁串行化；Worker 在每个扫描周期开始时重新加载一个完整快照。两个容器共享 `/data`，因此不要求 Worker 重启即可生效。
- Worker 成功投递后，先更新内存去重集合，再原子写回 `monitor.json`；重启后从该集合恢复，不会重新发送已持久化的键。新配置或参数同步不会回补历史标记：Worker 仅处理同步后新收盘的 K 线。
- 纯 JSON 无法把“外部企微已接收”和“本地写入已发送键”做成单一原子事务：在两者之间崩溃时，最小实现可能重复一次。选择成功后落键以优先避免漏通知；要消除这个窗口需恢复已排除的持久化投递账本。

### 一次性同步 API

- `POST /api/monitor/sync`：页面将当前 `marketType`、`interval`、`limit`、五个公式字段，以及通知开关为 `on` 的 `state.symbols` 作为一个完整 JSON 请求体提交。服务端校验、原子写 `/data/monitor.json`，并返回写入后的脱敏完整配置和 `updated_at`。
- `GET /api/monitor/status`：直接返回 `/data/worker-status.json`；文件不存在时返回 `state: "waiting_for_sync"`，不创建默认监控。
- `/api/wecom/notify` 继续只用于页面“测试通知”，不影响 `sent_signal_keys`。
