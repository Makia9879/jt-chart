## 2. 三层门禁：不可互相代替

| 层级 | 运行时机 | 网络 | 目的 | 失败后果 |
| --- | --- | --- | --- | --- |
| 普通 CI | 每次提交/PR | 默认断网，全用固定 fixture | 证明算法、解析、协议、缓存与竞态的确定性 | 不可合并 |
| Pixel 9 Pro XL 虚拟机发布门禁 | 候选版、发布前 | 先 fixture/可控故障，再单独公网 smoke | 证明 Android/WebView 集成、触摸、生命周期、断网恢复、签名与覆盖升级 | 不可生成可交付候选版 |

## 6. 断网、恢复、后台、旋转与进程重建

### 6.1 断网/恢复的 ADB 机械步骤

断网前先用成功 query 写入缓存，记录当前图表身份和更新时间。API 36 Pixel 优先使用按包名的网络拒绝，不影响虚拟机中其他 App：

```bash
PKG=com.makia.jtchart
adb -s "$SERIAL" shell cmd connectivity set-package-networking-enabled false "$PKG"
adb -s "$SERIAL" shell cmd connectivity set-chain3-enabled true
adb -s "$SERIAL" shell cmd connectivity get-package-networking-enabled "$PKG"
adb -s "$SERIAL" shell dumpsys connectivity > "$EVIDENCE/connectivity-off.txt"
adb -s "$SERIAL" exec-out screencap -p > "$EVIDENCE/offline-cache-bubble.png"

adb -s "$SERIAL" shell cmd connectivity set-package-networking-enabled true "$PKG"
adb -s "$SERIAL" shell cmd connectivity set-chain3-enabled false
adb -s "$SERIAL" shell cmd connectivity get-package-networking-enabled "$PKG"
adb -s "$SERIAL" shell dumpsys connectivity > "$EVIDENCE/connectivity-restored.txt"
```

必须断言：断网时缓存图和时间不变，错误归类 `Connectivity`，气泡持续且点击可重试；恢复后成功 query 原子替换图和时间，气泡消失。无论测试成功或失败，收尾都必须恢复包网络并关闭 chain。如未来系统镜像不提供按包名控制，才使用 `cmd connectivity airplane-mode enable/disable` 或设备网络面板，但必须保留原状态并留屏幕录像。用固定本地故障夹具分别覆盖 20 秒无响应和 HTTP 429，不等待公网偶发故障。

### 6.2 旋转

```bash
adb -s "$SERIAL" shell wm user-rotation lock 1
adb -s "$SERIAL" exec-out screencap -p > "$EVIDENCE/rotate-landscape.png"
adb -s "$SERIAL" shell wm user-rotation lock 0
adb -s "$SERIAL" exec-out screencap -p > "$EVIDENCE/rotate-portrait-restored.png"
adb -s "$SERIAL" shell wm user-rotation free
```

旋转前必须打开抽屉并修改未应用草稿，将图表平移到历史区间。断言：旋转不增加网络请求数或 `fetchedAt`，草稿、requested/displayed 身份、置顶气泡、数据和匹配 logical range 都恢复；不同 fingerprint 不恢复旧 range。

### 6.3 后台/锁屏与进程重建

```bash
adb -s "$SERIAL" shell input keyevent KEYCODE_HOME
adb -s "$SERIAL" shell am kill com.makia.jtchart
adb -s "$SERIAL" shell pidof -s com.makia.jtchart
adb -s "$SERIAL" shell monkey -p com.makia.jtchart -c android.intent.category.LAUNCHER 1

adb -s "$SERIAL" shell input keyevent KEYCODE_SLEEP
adb -s "$SERIAL" shell input keyevent KEYCODE_WAKEUP
adb -s "$SERIAL" shell wm dismiss-keyguard
```

两类恢复分别验收：

- **仅进入后台/锁屏**：活动请求与 timer 取消，期间请求计数不增长；回前台只立即刷新 1 次，不补跑错过周期。
- **进程重建**：先应用设置、写入缓存和 viewport，再打开但不应用一份草稿；`am kill` 后重启，已应用设置/缓存/匹配 viewport 恢复，草稿丢弃。再以 `am force-stop` + 重启重复持久化验收。

`am kill` 前 App 必须在后台；如 shell 回报未杀死或 PID 未变，该次证据无效，必须重跑，不能只凭界面看似恢复就判定通过。

## 7. WebView 安全与无后端证明

### 7.1 自动化负测试

使用打包进 test APK 的本地恶意探针页，而不在 release 暴露调试菜单。必须证明：

- `https://appassets.androidplatform.net/assets/chart/index.html` 是唯一允许的顶层页；`file://`、`content://`、其他 appassets 页和所有 `http(s)` 外部导航被拒；
- CSP 下 `fetch`/XHR/WebSocket、远程 script/style/frame/object 失败，因此只有 Kotlin Repository 能联网；
- `allowFileAccess=false`、`allowContentAccess=false`、DOM storage 关闭、三方 Cookie 关闭、mixed content `NEVER`；
- 新窗口、下载、地理位置/摄像头/麦克风权限请求被拒；
- 每次页面重建销毁旧 message port，旧 page instance 的回调无效；
- release 的 manifest `debuggable=false`，且仪器化/代码审计证明 `WebView.setWebContentsDebuggingEnabled(false)` 在 release 不可被 debug 开关覆盖。

### 7.2 静态产物检查

- 解包/Android Lint 检查不含 CDN URL、旧 `/api/wecom/*`、`/api/monitor/*`、自建服务地址、`addJavascriptInterface` 或整批数据的 `evaluateJavascript` 路径。
- 六个适配器只允许四个官方上游 host：`data-api.binance.vision`、`fapi.binance.com`、`api.bybit.com`、`api.bitget.com`；本地页面另允许虚拟 origin `appassets.androidplatform.net`。公网 smoke 以设备网络完成，不启动 Docker、`chart_server.py` 或工作区 HTTP server。
- 冷启动前停止工作区与本 App 相关的临时服务；这是运行前置，不作为“APK 不依赖后端”的唯一证据。主证据是 release APK 静态检查 + 设备上六源直连。

## 8. 六数据源公网 smoke（不进普通 CI）

对 `BTCUSDT / 1h` 逐源串行运行，并在源之间留出契约的 host 起始间隔。分两类证据：

1. **端点探针**：按已关闭六源契约使用 `limit=3`，只验证当时的 HTTP/envelope/少量解析，该探针不绕过生产 `Query` 校验注入 App；
2. **App Repository 端到端**：使用生产合法最小值 `limit=120` 逐源请求，确认 App 实际适配器、缓存和图表全链路。

每源必须断言：

- HTTP 2xx 且 Binance 顶层数组 / Bybit `retCode=0` / Bitget `code="00000"`；
- 无认证头，得到非空 K 线，规范化后开盘时间严格递增、无重复，OHLC 为正且 high/low 关系成立；
- 时间落在合理的小时边界；连续两次时只允许最新未闭合根变化；
- 图表标题显示实际 source/symbol/interval/limit 和更新时间；切源时上一图保留到新源成功。

六源全通才是完整公网 smoke 通过。某源因地区或上游故障失败时，保留错误分类、HTTP/厂商 code（脱敏）、设备时间和网络类型证据，候选版不判通过；不修改基址、不加代理、不降级 HTTP。

| Source | Host / path | 证据名前缀 |
| --- | --- | --- |
| Binance Spot `spot` | `data-api.binance.vision/api/v3/klines` | `smoke-binance-spot-*` |
| Binance USDT-M `futures` | `fapi.binance.com/fapi/v1/klines` | `smoke-binance-futures-*` |
| Bybit Spot `bybitSpot` | `api.bybit.com/v5/market/kline?category=spot` | `smoke-bybit-spot-*` |
| Bybit Linear `bybitLinear` | `api.bybit.com/v5/market/kline?category=linear` | `smoke-bybit-linear-*` |
| Bitget Spot `bitgetSpot` | `api.bitget.com/api/v2/spot/market/candles` | `smoke-bitget-spot-*` |
| Bitget USDT Perp `bitgetFutures` | `api.bitget.com/api/v2/mix/market/candles` | `smoke-bitget-futures-*` |

不归档完整公网 response body 或整批 K 线；只留 source、请求参数、时间、结果计数、首尾 openTime、规范化校验结果、错误类别和必要的 HTTP/厂商 code。

```bash
apkanalyzer manifest version-code "$APK" > "$EVIDENCE/version-code.txt"
apkanalyzer manifest min-sdk "$APK" > "$EVIDENCE/min-sdk.txt"
apkanalyzer manifest target-sdk "$APK" > "$EVIDENCE/target-sdk.txt"
apkanalyzer manifest debuggable "$APK" > "$EVIDENCE/debuggable.txt"
shasum -a 256 "$APK" > "$EVIDENCE/apk-sha256.txt"
```

断言 application ID `com.makia.jtchart`，首版 `versionName=0.1.0`、`versionCode=1`，min SDK 29，target SDK 等于项目锁定值，`debuggable=false`，签名证书 SHA-256 等于首发基线，APK SHA-256 与 `SHA256SUMS.txt` 一致。当前 SDK 缺 `cmdline-tools/latest`/`apkanalyzer` 时，这是 release 门禁的环境阻塞，不得删掉检查项或口头核对。

### 9.2 安装与覆盖升级

```bash
adb -s "$SERIAL" install "$BASELINE_APK"
# 在基线版中应用非默认设置，并让至少一个 query 成功写入缓存
adb -s "$SERIAL" install -r "$APK"
adb -s "$SERIAL" shell monkey -p com.makia.jtchart -c android.intent.category.LAUNCHER 1
adb -s "$SERIAL" shell dumpsys package com.makia.jtchart > "$EVIDENCE/package-after-upgrade.txt"
```

首个正式版做全新安装并留下可重复安装的基线 APK；第二个及以后版本必须从“上一个已发布 release”真实 `adb install -r` 覆盖。断言新 PID 可启动、版本正确、非默认设置/最后成功缓存/viewport 保留。正常升级门禁不允许先 `adb uninstall`、清数据或用 debug APK 代替。

发布目录只含 APK、`SHA256SUMS.txt`、`RELEASE_NOTES.md`；不含本验收证据、keystore、密码、`.env` 或 Gradle 秘密。

## 10. 证据命名、清单与留存

### 10.1 目录和命名

每次门禁创建独立目录，UTC 时间不使用空格：

```text
android-app/qa-evidence/{versionName}-c{versionCode}/{RUN_ID}-{device}/
RUN_ID = YYYYMMDDTHHMMSSZ-{shortGitSha}
device = ci | pixel9proxl-api36 | xiaomi17promax-hyperos
```

单文件：

```text
{gateId}-{scenario}-{orientation}-{result}.{ext}
```

例：`PX-BUBBLE-01-new-query-failure-portrait-pass.png`、`smoke-bybit-linear-summary-pass.json`。每个目录必须有 `manifest.json` 记录 commit、dirty/clean 状态、版本、设备 serial 的脱敏别名、Android/API/ABI/WebView 版本、开始结束时间、门禁 ID、pass/fail/skip 及跳过理由，并列出各证据文件 SHA-256。`commands.txt` 保留实际命令和退出码，但先删除签名 secret 环境和用户绝对私有路径。

### 10.2 必须证据

- CI：JUnit XML/HTML、Lint 报告、Gradle 任务结果、fixture/golden 变更 diff。
- Pixel：基线 `getprop/wm/WebView` 摘要、instrumented 报告、关键截图/录像、脱敏 Logcat、请求/generation/viewport 计数摘要、六源 smoke 摘要。
- Release：`apksigner`、manifest 六项、signer 指纹、APK SHA-256、覆盖安装输出、升级前后设置/缓存证据。
- 真机：设备/HyperOS/系统 WebView 版本、安装器屏幕录像、竖横屏和安全区截图、六源结果、断网/锁屏/恢复、覆盖升级证据。

### 10.3 留存规则

- `android-app/qa-evidence/` 必须 gitignored；证据不进公开 APK 分发包，不与签名秘密同库保存。公共 `android-app/dist/<release>/` 只含 APK、`SHA256SUMS.txt`、`RELEASE_NOTES.md`。
- 失败的 CI/Pixel 证据至少保留 30 天，便于回归；与已关闭缺陷关联的失败证据随缺陷记录留存。
- 每个对外发布版的完整 Pixel + 真机 + 签名/升级证据与该 release APK 同寿命留存；如没有永久制品库，则发布阻塞而不是临时放在开发机。
- 非发布的通过证据可在 30 天后清理，但 fixture、golden 基线和机器可读测试报告摘要应由 CI 制品策略可追溯。

## 11. 小米 17 Pro Max / HyperOS 真机最终发布关卡

真机关卡只在设备在线、用户允许 ADB 调试和候选 release APK 已通过 Pixel 门禁后执行。使用真实签名 release，不用 debug/test-only fixture build 代替。

| ID | 场景 | 必须结果 | 证据 |
| --- | --- | --- | --- |
| HW-INSTALL-01 | HyperOS 安装器全新安装 | 校验来源提示后可安装、首启动无 crash，包名/版本/签名正确 | 安装录像 + package/apksigner 摘要 |
| HW-UPGRADE-01 | 上一 release → 候选 release 覆盖安装 | 不卸载/不清数据；已应用设置、缓存、viewport 保留 | 升级前后录像/截图 + 安装输出 |
| HW-INSET-01 | 竖屏/横屏、状态栏/导航栏/挖孔安全区 | 标题、置顶气泡、抽屉和图表不遮挡/裁切，触摸区不与系统返回手势冲突 | 全屏截图 + 录像 |
| HW-GESTURE-01 | 抽屉左缘、系统返回、图表缩放/平移 | 12dp 抽屉边界和遮罩规则可用，WebView 无误抢/丢手势 | 触摸录像 |
| HW-WEBVIEW-01 | 设备实际 Android System WebView | 本地资源/HELLO/READY/绘图正常，安全负测试的产品可观察结果不退化 | WebView 版本 + 脱敏 Logcat + 截图 |
| HW-NET-01 | Wi-Fi 六源 smoke | 六源全链路成功，无后端/代理 | 六源摘要 |
| HW-NET-02 | 移动网络（如设备/SIM 可用） | 六源或已记录的运营商限制；限制不伪装成 App 成功 | 网络类型 + 六源摘要 |
| HW-OFFLINE-01 | Wi-Fi/数据全关→恢复 | 旧图和时间保留，气泡正确；恢复后原子刷新 | 录像 + 前后截图 |
| HW-LIFE-01 | 后台、HyperOS 锁屏、系统回收后恢复 | 后台无 timer/请求；回前台仅一次刷新；持久化恢复正确 | 请求计数 + 录像 |
| HW-ROTATE-01 | 带草稿/历史 viewport 反复旋转 | 无重请求、无空白页、无视野跳回，草稿在配置变更中保留 | 录像 + 请求计数 |
| HW-STABILITY-01 | 连续使用 30 分钟，轮换六源/周期/旋转/前后台 | 无 crash/ANR/显著空白，无并发刷新或旧 generation 回写 | 录像摘要 + bugreport/Logcat（脱敏） |

移动网络不可用时，`HW-NET-02` 可记为有理由的 `skip`，不影响 Wi-Fi 六源必过。其他真机必过项不允许以“Pixel 已过”跳过。真机关卡失败后修复需重跑受影响项及一次完整 smoke，不在现场改配置或网络边界规避。

## 12. 发布判定

候选版只有在以下条件全部成立时才可交付：

1. 普通 CI 全绿，fixture/golden 无未审批更新；
2. Pixel 9 Pro XL 的 instrumented/UI、断网/恢复、旋转、进程重建、WebView 安全和六源 smoke 全通；
3. release APK 签名、manifest、SHA-256、非调试、无 secret 检查通过；第二版起真实覆盖升级通过；
4. 小米 17 Pro Max / HyperOS 真机必过项全通；
5. 证据 `manifest.json` 完整、文件哈希可核对，无未解释 skip，且永久发布证据已入制品库。

如真机尚未在线，可将 Pixel 通过的 APK 标记为“开发候选版”，但不得标记为“最终发布通过”。
