# 缓存、前台刷新与状态恢复契约

## 目标

`JT Chart` 在首次启动、无网、请求失败、快速切换、旋转、后台恢复和进程重建时，必须优先保留可用图表，同时不得把旧数据伪装成新数据。

## 持久化分工

### 已应用设置

- 使用单个类型化 Proto DataStore 保存 schema version、币种列表及顺序、当前币种、数据源、周期、K 线数量、全部算法参数、自动刷新开关与间隔。
- DataStore 只写入通过校验且用户点击“应用并刷新”后的设置。写入采用原子 update，不拆成多个可部分成功的文件。
- 抽屉未应用草稿只存在 ViewModel：横竖屏旋转保留，App 关闭或进程被回收后丢弃；不写 DataStore、Room 或 `SavedStateHandle`。
- DataStore 读取/迁移失败时使用已固化的默认值并显示本地配置错误，不阻断用户重新设置。

### 最后成功 K 线快照

- 使用 Room 保存规范化 K 线快照，而不是 WebView storage 或 `Bundle`。
- 缓存键是严格查询键 `source + symbol + interval + limit`。算法参数不影响原始 K 线键；修改算法参数直接用已缓存 K 线重算。
- 每个键只保留一份最新成功快照：`queryKey`、`fetchedAtEpochMs`、`lastAccessedAtEpochMs`、`datasetFingerprint`、原始 DecimalString K 线数组和协议/schema version。
- 只有网络成功且全部规范化校验通过后才在单个 Room transaction 中替换快照。取消、超时、HTTP/厂商错误、协议错误和空数据都不覆盖旧快照。
- 缓存无时间过期删除：无论多旧都可显示，但必须同时显示准确更新时间并尝试刷新。
- 容量同时受两个上限约束：最多 64 个查询快照，数据库中快照 payload 合计最多 32 MiB。任一上限超出时按 `lastAccessedAt` LRU 清理，当前展示键与正在写入的键不清理。
- 数据库损坏时可丢弃并重建行情缓存，但不同时删除 DataStore 用户设置；重建事件进入本地诊断。

### 图表 viewport

- viewport 保存 `{queryKey, datasetFingerprint, logicalFrom, logicalTo, updatedAt}`，不保存像素坐标、十字光标或完整 K 线。
- ViewModel 保留当前 viewport 用于旋转；Room 保留每个快照最后 viewport 用于进程重建。JS 变更回报以 500 ms debounce 写入，并在页面停止前尽力 flush。
- 仅 `queryKey + datasetFingerprint` 完全匹配时恢复 logical range；不匹配时 `fitContent()`。越界 range 夹到当前数据域。

## 可见状态与失败气泡

### 数据身份

- 原生 UI 同时维护 `requestedQuery`和 `displayedDataset`，二者不得混为一个标题。
- 图表标题始终显示实际 `displayedDataset` 的币种、数据源、周期、K 线数量与更新时间。
- 新查询尚未成功时，保留上一张成功图表，不把图表标题伪装为新查询。

### 顶部小型置顶气泡

- 新查询失败时，顶部显示 App 内小型置顶气泡，不使用几秒后消失的 Android Toast。
- 气泡必须同时说明失败的 `requestedQuery` 和当前仍在显示的 `displayedDataset`，例如：“ETH / Bybit / 4h 请求失败，当前仍显示 BTC / Binance / 1h 数据”。
- 气泡持续置顶，直到该请求成功或用户切换到另一查询；不自动超时消失。点击气泡立即重试当前 requestedQuery。
- 加载阶段可用同一顶部区域显示“正在加载 requestedQuery，下方为 displayedDataset”，但不遮挡图表操作。

### 缓存新鲜度

- 不使用模糊的“最新”掩盖缓存年龄。缓存图表始终显示 `缓存数据 · 更新于 YYYY-MM-DD HH:mm:ss`，可辅以“N 分钟/天前”。
- 当前会话成功网络刷新后显示“已更新 · HH:mm:ss”。请求失败时保留快照时间不动，不用失败时间替代数据时间。
- 无论缓存多旧都可显示并立即尝试刷新；不设一个超过后就隐藏数据的时间 TTL。

## 主要场景

### 首次启动与进程重建

1. 读取已应用设置，形成 requestedQuery。
2. 如有完全匹配的查询缓存，立即绘制并显示缓存时间，同时发起网络刷新。
3. 如无同键缓存且进程内无上一张图，显示加载空态；失败后显示错误空态和置顶重试气泡。
4. 进程重建只从 DataStore/Room 恢复已应用设置、快照和匹配 viewport，不恢复抽屉草稿。

### 新查询和失败

- 提交新查询时，先查同键缓存：有则立即替换为该缓存并刷新；无则保留上一张成功图并显示加载气泡。
- 新查询失败且无同键缓存时，继续显示上一张成功图，并用气泡明确两个数据身份。
- 任何失败都不清空已成功图、不覆盖缓存、不伪造更新时间。

### 快速连续切换

- 每次选择递增 generation，取消上一网络请求、重试、延时 timer 和未发送图表快照。
- requestedQuery 始终是最后一次选择；只有最新 generation 能更新 UI、Room 和 WebView。
- 旧请求即使无法及时物理取消，其迟到结果也必须在写缓存前再校验 generation 并丢弃。
- 加载气泡只显示最终选择，不排队显示中间选择。

### 自动刷新与视野

- 使用一次性 timer，每次请求完成或失败后从当前时刻重新计算下次刷新，不使用可堆叠的 fixed-rate 调度。
- 只有 App 处于前台、屏幕未锁定、图表可见、WebView `READY`、当前无活跃请求且自动刷新开启时才调度。
- 自动刷新替换同一 dataset 后：用户在历史位置时保持当前 logical range；用户原本位于最右侧最新 K 线附近时才继续跟随最新数据。
- 应用新币种、周期、数据源或数量后，对新 dataset `fitContent()`。仅修改算法参数时保持当前 viewport。

### 后台、锁屏和恢复前台

- 进入后台、锁屏或图表不可见时，取消自动刷新 timer 和已绑定页面的活跃请求；不在后台创建 Worker/Alarm。
- 恢复前台且图表可见时立即刷新一次，不补跑离开期间错过的每个周期。
- 恢复刷新成功或失败后，再从当前时刻计算下次自动刷新。
- 如恢复时 WebView 正在重建，先用内存/持久化快照完成 `READY → RENDER_SNAPSHOT`，再发起一次网络刷新，避免空白首屏。

### 旋转

- 旋转不发起网络请求、不更新缓存时间、不重置自动刷新计时。
- ViewModel 保留已应用设置、未应用草稿、requestedQuery、displayedDataset、K 线快照、顶部气泡状态和 viewport。
- 新 WebView `READY` 后先原子重发 displayedDataset，再恢复匹配 logical range。

## 竞态与不变量

- 同一图表最多一个活跃行情请求；自动、手动、前台恢复和设置应用都走同一个请求协调器。
- 网络响应、Room 写入、WebView 发送和 JS 回执前都重新校验 page instance/generation/query key。
- `Cancelled` 不显示为用户错误；其他失败进入稳定错误气泡并保留数据。
- 手动点击重试/刷新时，如同一 query 已有请求运行，不并发第二个；可将当前请求标记为用户显式重试。

## 验收点

- 冷启动时同键缓存先显示、后刷新；7 天以上缓存仍显示准确时间。
- 新查询失败保留旧图，气泡同时显示 requested/displayed 身份，持续到成功或切换，点击可重试。
- 快速切换只最后 generation 生效，迟到响应不写 UI/缓存。
- 自动刷新不并发；历史视野保持，最新边缘继续跟随。
- 后台无请求和计时器；恢复时只刷新一次。
- 旋转不联网且保留草稿/图表/视野；进程重建丢弃草稿但恢复已应用设置、缓存和匹配视野。
- 缓存 LRU 同时满足 64 快照/32 MiB 上限，失败不覆盖成功快照。
