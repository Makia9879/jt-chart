# 六数据源直连契约

> 票据：[定义六数据源直连契约](../tickets/0002-定义六数据源直连契约.md)
> 核对日期：2026-07-14
> 范围：Android App 直接读取公开 K 线；不使用 API Key、签名、自建后端或代理。

## 结论

首版用六个固定适配器，不提供用户可编辑的基址。六个适配器都只接受 App 的统一参数 `symbol`、`interval`、`limit`，输出同一份按开盘时间严格递增的 K 线列表。

统一输入：

- `symbol`：去除首尾空白后转大写，只允许 `[A-Z0-9]+`。首版币对格式保持现有 `BTCUSDT` 形式。
- `interval`：只允许 `1m | 5m | 15m | 1h | 4h | 1d | 1w`。
- `limit`：`120..1000`，默认 `500`。这是六源共同上限，也保持现有图表的 WMA 计算边界。
- 首屏和自动刷新不传起止时间，读取最近 `limit` 根；以后若做历史翻页，时间参数必须仍用 Unix 毫秒。

统一输出 `Candle`：

```text
openTimeMs: Long
open: DecimalString
high: DecimalString
low: DecimalString
close: DecimalString
baseVolume: DecimalString?
quoteVolume: DecimalString?
```

- `openTimeMs` 在 Kotlin 中用 `Long`，传给 Lightweight Charts 时再整除 `1000` 得 Unix 秒。
- 所有价格、成交量先以原始十进制字符串保存和缓存，并用 `BigDecimal` 做格式/范围校验；不要在网络层或缓存层转成 `Float`/`Double`。只有交给 JavaScript 绘图和现有指标算法时才转为 JS `Number`，从而保留价格位数推断所需的尾随零。
- 适配器必须按 `openTimeMs` 升序排序、按时间戳去重，并拒绝非有限数、非正价格、`high < max(open, close)`、`low > min(open, close)` 等畸形行。上游少于请求数量但非空是有效响应；空列表归类为 `NoData`。
- 最新一根可能尚未闭合，仍然保留，行为与现有图表一致；界面不能把它标成已闭合数据。

## 六源请求与解析

| App source | 请求 | 固定参数 | 区间映射 | 上游数量 | 上游顺序与解析 |
|---|---|---|---|---|---|
| `spot` | `GET https://data-api.binance.vision/api/v3/klines` | 无 | 原值 | 默认 500，最大 1000；权重 2 | Binance REST 总则声明默认按时间正序。数组 `[openTime, open, high, low, close, volume, closeTime, quoteVolume, ...]`，取 0–5、7。 |
| `futures` | `GET https://fapi.binance.com/fapi/v1/klines` | 无 | 原值 | 默认 500，最大 1500；本 App 封顶 1000 | 字段位置同 Spot。该端点页未单独承诺排序，必须显式排序。权重按 limit：`[1,100)=1`、`[100,500)=2`、`[500,1000]=5`、`>1000=10`。 |
| `bybitSpot` | `GET https://api.bybit.com/v5/market/kline` | `category=spot` | 见下表 | 默认 200，范围 1–1000 | `retCode` 必须为 0；`result.list` 明确按 `startTime` 倒序，解析 `[startTime, open, high, low, close, volume, turnover]` 后反转/统一排序。 |
| `bybitLinear` | 同上 | `category=linear` | 见下表 | 同上 | 同 Bybit Spot；USDT 合约的 volume 是 base coin，turnover 是 quote coin。 |
| `bitgetSpot` | `GET https://api.bitget.com/api/v2/spot/market/candles` | 无 | 见下表 | 默认 100，最大 1000 | `code` 必须为 `00000`；解析 `data` 的 `[timestamp, open, high, low, close, baseVolume, usdtVolume, quoteVolume]`。官方页面未承诺列表顺序，必须显式排序。 |
| `bitgetFutures` | `GET https://api.bitget.com/api/v2/mix/market/candles` | `productType=USDT-FUTURES`、`kLineType=MARKET`（可省略但应显式） | 见下表 | 默认 100，最大 1000 | `code` 必须为 `00000`；解析 `[timestamp, open, high, low, close, baseVolume, quoteVolume]`。官方示例为正序但未定义顺序，必须显式排序。 |

所有请求都带 `symbol` 和 `limit`。Binance/Bybit 的区间参数名为 `interval`；Bitget 为 `granularity`。

| App interval | Binance Spot / USDT-M | Bybit Spot / Linear | Bitget Spot | Bitget USDT-FUTURES |
|---|---:|---:|---:|---:|
| `1m` | `1m` | `1` | `1min` | `1m` |
| `5m` | `5m` | `5` | `5min` | `5m` |
| `15m` | `15m` | `15` | `15min` | `15m` |
| `1h` | `1h` | `60` | `1h` | `1H` |
| `4h` | `4h` | `240` | `4h` | `4H` |
| `1d` | `1d` | `D` | `1day` | `1D` |
| `1w` | `1w` | `W` | `1week` | `1W` |

Bitget 对部分细粒度区间限制可查询的历史跨度；首版只读最近至多 1000 根，不做超跨度分页。Mix 的 `startTime`/`endTime` 查询窗口最大 90 天；这不影响首版的最近数据请求。

## 超时、并发与限频

以下是 App 契约，不是假称交易所官方默认值：

- OkHttp：连接超时 10 秒、读取超时 15 秒、单次调用总超时 20 秒。
- 同一图表只允许一个活跃行情请求。切换币种/周期/数据源或页面离开前台时取消旧请求；被取消的请求不得更新 UI 或缓存。
- 自动刷新最短仍为现有的 15 秒。快速手动操作时，同一 host 的请求起始时间至少间隔 250 ms；这远低于各官方上限且避免无意义并发。
- 对连接失败、读取超时、HTTP `408`、`429`、`5xx`，最多自动重试 1 次；指数退避基数 1 秒并加随机抖动。若响应有 `Retry-After` 或厂商重置时间，优先遵循；没有时 429 至少等 5 秒。
- Binance `418` 是持续违反 429 后的 IP 自动封禁，不自动重试。Bybit 的 `403, access too frequent` 要终止该 host 的 HTTP 会话并至少等待 10 分钟，不自动轮询。
- 普通 `4xx`、厂商参数/币对错误、协议形状错误和 `NoData` 不自动重试。前台恢复触发的新一轮刷新不计为同一次重试。

官方限频依据：

- Binance Spot Kline 单次 IP 权重 2；USDT-M Kline 权重随 limit 变化，App 最大 1000 时最多权重 5。Binance 用 HTTP 429 表示超限，忽略 429 继续请求可能返回 418。
- Bybit 默认 IP 上限为每 5 秒 600 次；超限返回 `403, access too frequent`，至少等待 10 分钟。响应体 `retCode=10006` 表示 API rate limit，响应头 `X-Bapi-Limit`、`X-Bapi-Limit-Status`、`X-Bapi-Limit-Reset-Timestamp` 可用于状态和退避。
- Bitget 两个 Candle 端点均标明每 IP 每秒 20 次。

## 错误模型

网络层只向 UI 暴露稳定分类，原始 HTTP 状态、厂商 code/retCode/msg 进入本地诊断字段，不直接决定用户文案：

| 分类 | 判定 | 是否保留缓存图 | 自动重试 |
|---|---|---:|---:|
| `Cancelled` | 生命周期或新请求取消 | 是 | 否 |
| `Connectivity` | DNS、TLS、无网络、连接失败 | 是 | 最多 1 次 |
| `Timeout` | connect/read/call timeout | 是 | 最多 1 次 |
| `RateLimited(retryAt?)` | HTTP 429/418；Bybit 403 频率封禁或 `10006`；明确的 Bitget 频率错误 | 是 | 仅普通 429 可按退避重试 1 次；418/Bybit 403 否 |
| `RequestRejected` | HTTP 400/404 或非成功厂商 code，且属于参数、区间、币对错误 | 是 | 否 |
| `AccessRestricted` | 非限频的 HTTP 401/403、地区/服务访问限制 | 是 | 否 |
| `UpstreamUnavailable` | HTTP 408/5xx 或明确的厂商系统繁忙错误 | 是 | 最多 1 次 |
| `Protocol` | 非 JSON、字段缺失、数值无法解析、成功 envelope 形状错误 | 是 | 否 |
| `NoData` | 成功 envelope 但列表为空或规范化后为空 | 是 | 否 |

成功条件必须同时满足 HTTP 2xx 和厂商 envelope：Binance 顶层为数组；Bybit `retCode == 0`；Bitget `code == "00000"`。未知的非零厂商 code 默认 `RequestRejected`，只有显式列入适配器的限频/系统错误码才能重分类，避免根据易变文案猜测。

日志禁止记录设备网络标识或未来可能加入的凭据；本契约的请求也不得添加 `X-MBX-APIKEY`、`X-BAPI-API-KEY`、`ACCESS-KEY` 等认证头。

| U07 | 7 个统一区间 × 6 数据源 | 42 个映射全部精确匹配上表，大小写敏感 |

| U10 | Bitget 乱序和重复时间戳 fixture | 排序并去重；结果稳定 |
| U11 | 18 位小数和尾随零 | 缓存/桥接前字符串完全一致，`BigDecimal` 可解析 |

| U17 | 请求头检查 | 六源都没有认证头、Cookie 或用户可注入 host |

### 公网 smoke（手动/发布验收，不进普通 CI）

对 `BTCUSDT`、`1h`、`limit=3` 逐个调用六源：

1. HTTP 和厂商 envelope 成功，无认证头。
2. 得到 1–3 根可解析 K 线；规范化后时间严格递增且无重复。
3. 每根 `openTimeMs` 可整除到合理的小时边界，OHLC 为正且高低关系成立。
4. 连续调用两次时，只有最新未闭合根允许价格/成交量变化；较早根不应改变。
5. 在 Pixel 虚拟机分别模拟断网、20 秒无响应、HTTP 429 fixture，确认缓存图仍显示且错误分类正确。

公网 smoke 只能证明当时网络和币对可用，不能替代 fixture，也不能因地区访问失败而改用自建代理；地区限制应呈现为 `AccessRestricted`，用户仍可切换其他数据源。

## 与现有实现的关系

现有 `jt_shared.py` 已经给出六源基址、固定参数和 7 组区间映射，并对 Bybit 倒序做了反转；这些可作为迁移输入。Android 实现不能直接照搬以下行为：

- `rows_to_candles` 当前立即转 Python `float`，不符合本契约的十进制字符串保真要求。
- Bitget 当前直接返回上游数组；新适配器必须对所有厂商统一排序和去重。
- 当前 `requests.get(..., timeout=15)` 只有单一超时和粗粒度异常；Android 必须采用上述稳定错误分类、取消和一次重试边界。
