## Question

首版需用哪些自动化测试、固定夹具、ADB 命令、虚拟机交互、断网/恢复、横竖屏、进程重建、六数据源实请求、图表对比和签名 APK 检查，才能证明纯 App 无后端运行且不破坏现有策略语义？需分别给出 Pixel 9 Pro XL 必过门禁和小米 17 Pro Max 设备上线后的最终发布关卡，并为每个验收项指定可保留的证据。

## Resolution

已形成[端到端验收矩阵](../assets/端到端验收矩阵.md)，明确区分普通 CI、Pixel 9 Pro XL 虚拟机发布门禁和小米 17 Pro Max / HyperOS 最终真机关卡。

关键结论：

- 普通 CI 禁止公网和签名秘密，用固定夹具覆盖算法、六适配器、错误/重试、generation 竞态、Room/DataStore、WebMessage 协议和 WebView 安全。
- Pixel 门禁在已启动的 API 36 ARM64 虚拟机上执行构建/安装/启动、手势/金图、定向断网/恢复、旋转、前后台、进程重建、六源 smoke、签名验签和覆盖升级。
- “无自建后端”需同时由 APK/源码四官方 host allowlist、WebView `connect-src 'none'` 以及不启动仓库服务时的设备六源直连证据证明。
- release 必须通过 `apksigner`、manifest/版本/SDK/`debuggable`、SHA-256、secret 文件扫描和保留数据的 `adb install -r` 覆盖升级。
- 小米真机未在线不阻塞开发候选版，但仍阻塞最终对外交付；需以同一 SHA-256 的 release APK 验收 HyperOS 安装器、安全区/手势、WebView、六源、生命周期和覆盖升级。
- 每个门禁按 run/device 留存命令退出码、JUnit/Lint、脱敏 Logcat、截图/录屏、六源摘要、验签/manifest/哈希和机器可读 `manifest.json`。

本 Resolution 只定义实施后必须通过的门禁；当前 `android-app/` 仍未创建，除虚拟机设备基线外，不声称任何 App/APK 验收已执行或通过。
