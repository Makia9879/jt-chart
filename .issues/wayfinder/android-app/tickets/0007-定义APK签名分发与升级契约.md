## Question

在不使用应用商店的前提下，`JT Chart` 应如何生成、命名、校验和交付可持续覆盖升级的 release APK？需根据 Android 官方签名机制明确 keystore 的存放、密码注入、Git 忽略、备份、versionCode/versionName、安装前校验与丢失密钥后果；不将密钥或密码写入仓库。

## Resolution

采用仓库外唯一 app signing key 签名所有 release APK；首版为 `versionName=0.1.0` / `versionCode=1`，后续每次分发严格递增 `versionCode`。签名参数仅通过仓库外 `$GRADLE_USER_HOME/gradle.properties` 或受控的 `ORG_GRADLE_PROJECT_*` 环境变量注入，keystore、密码和私钥不进 Git 也不进分发包。交付文件命名为 `JT-Chart-v{versionName}-c{versionCode}-release.apk`，必须用 `apksigner`、`apkanalyzer` 和 SHA-256 校验，并在 Pixel 9 Pro XL 上通过 `adb install -r` 验证保留数据的覆盖升级。密钥应做异地加密备份和恢复演练；不使用 Play App Signing 时，丢失密钥就无法再覆盖更新原 application ID。

完整契约与官方来源：[APK 签名、分发与升级契约](../assets/APK签名分发与升级契约.md)。
